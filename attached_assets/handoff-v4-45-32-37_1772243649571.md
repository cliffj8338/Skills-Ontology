## Executive Summary

This session rebuilt the skill matching and extraction pipeline from a basic keyword matcher into a 6-pass ontology-aware matching engine. The work also introduced an admin skill blocklist (Firestore-backed) for manual curation of extraction noise, and unified filtering across both the Jobs tab and Blueprint Advantage Compare engine.

### 2. Skill Extraction Pipeline Overhaul (v4.45.32-33)

**Problem:** `parseJobLocally()` had three passes but only checked the old O*NET library (139 skills). The 43K skill library (ESCO + Lightcast + O*NET + Trades) was underutilized. Short skill names caused false-positive substring matches ("plan" matching "planning", "sing" matching "processing").

**Changes to `parseJobLocally()`:**

- **Pass 1 (user skills):** Added word-boundary matching for skills ≤5 chars. Prevents "plan" from matching "planning".
- **Pass 2 (skill library):** Now searches all 43K skills with word-boundary matching for terms ≤4 chars (SQL, CRM, etc.).
- **New synonym pass:** Checks `_synonymLookup` (670+ mappings) after library matching.
- **Validation layer (v4.45.33):** Before returning skills, runs through:
  - `SKILL_BLOCKLIST` (~80 common English words/fragments)
  - `VALUES_BLOCKLIST` (~25 personality traits that aren't skills)
  - Fragment detection (≤4 chars without vowels, not known acronyms)
  - Single words ≤6 chars verified against skill library
  - Metrics regex (gross margin, EBITDA, revenue, etc.)
  - Deduplication by normalized name
  - Admin blocklist filter

**Location:** `parseJobLocally()` function, starts ~line 30700.

### 3. Match Deduplication (v4.45.33)

**Problem:** "Management Mastery" appeared 4x in matched skills because "Management Consulting", "Knowledge Management", "Change Management", and "Management" each substring-matched the user skill "Management" separately.

**Fix:** After matching loop, deduplicate by user skill name, keeping highest-credit match. Also deduplicate gaps by normalized name. Recalculate earnedWeight from deduped matches.

**Location:** `matchJobToProfile()` function, after the main `jobSkills.forEach()` loop.

### 4. Six-Pass Matching Engine (v4.45.34, v4.45.36)

The matching engine in `matchJobToProfile()` now runs 6 passes before declaring a gap:

| Pass | Method | Credit | Added In | Location |
|------|--------|--------|----------|----------|
| 1 | Exact name / synonym | 100% | Original | `userMap[jLower]` |
| 2 | Substring containment | 85% | Original | `jLower.indexOf(key)` |
| 3 | Word overlap ≥50% | up to 75% | Original | `userNameIndex.forEach` |
| 4 | ESCO subcategory sibling | 55% | v4.45.34 | `getEscoCategorySiblings()` |
| 5 | Concept-root stems | up to 45% | v4.45.36 | `_stemEquiv` decomposition |
| 6 | Seniority domain implication | 40% | v4.45.36 | Keyword domain inference |

**Pass 4 — ESCO Subcategory Siblings:**
- Uses `_escoSubcategoryIndex` (built at load from 43K library)
- Capped at subcategories with ≤25 skills to prevent overly broad matches
- Fuzzy subcategory resolution: if exact name not in bridge, tries substring match against bridge keys
- Requires user skill at Advanced+ (proficiency ≥ 3)
- `matchType = 'sibling'`

**Pass 5 — Concept-Root Stems:**
- Decomposes both gap skill and user skills into stemmed word roots via `_stemEquiv`
- Checks 5-character prefix overlap between roots
- Single-word skills require 100% root match; multi-word require ≥40%
- Requires user skill at Advanced+
- `matchType = 'concept'`

**Pass 6 — Seniority Domain Implication:**
- Only activates for Executive/Senior users (detected from `userData.roles`)
- Infers gap skill domain using keyword patterns across 8 domains: business, finance, analytics, strategy, management, HR, technology, marketing
- Checks if user has Expert+ (Executive) or Mastery (Senior) in same domain
- Domain inference works via both ESCO bridge AND keyword fallback
- `matchType = 'implied'`

**Location:** `matchJobToProfile()` function, between pass 3 and `if (found)` block, ~lines 32580-32800.

**Category bridge enhancement (v4.45.34):** `buildEscoCategoryIndex()` now indexes user's own profile skills alongside the 43K library, enabling reverse lookups.

### 5. Admin Skill Blocklist (v4.45.35)

**Firestore-backed admin tool for manually curating skill extraction.**

**Data Model:**
- Firestore path: `meta/skillBlocklist`
- Schema: `{ skills: string[], updatedAt: string }`
- Runtime: `_adminSkillBlocklist` (Set)

**Functions:**
- `loadAdminSkillBlocklist()` — loads from Firestore on auth
- `saveAdminSkillBlocklist()` — persists to Firestore
- `isSkillBlocklisted(name)` — checks membership (lowercase normalized)
- `adminBlockSkill(name, jobIdx)` — adds to blocklist, re-renders job if viewing
- `adminUnblockSkill(name)` — removes from blocklist
- `renderAdminBlocklistPanel()` — renders management UI
- `adminBlockSkillFromInput()` — handles manual text input

**UI Integration:**
- Gap skill chips show ✕ button (admin-only, checks `fbIsAdmin`)
- Admin → Config tab has "Skill Blocklist" panel at bottom
- Panel shows all blocked skills as red chips with unblock button
- Text input for manual additions
- Toast confirmation on block/unblock

**Pipeline Integration:**
- Checked in `parseJobLocally()` validation layer
- Checked in `matchJobToProfile()` gap deduplication
- Checked in `_wbSkillQualityFilter()` (Compare engine)
- Loads for ALL users on auth (not just admin) so blocklist affects all parsing

**Location:** Functions defined ~line 16420. UI in `renderAdminConfig()` ~line 4380. Gap chip buttons in job detail rendering ~line 33158.

### 6. Compare Engine Filter Unification (v4.45.37)

**Problem:** Blueprint Advantage Compare had its own `_wbSkillQualityFilter()` with a weaker blocklist that didn't check admin blocklist. "Ashing", "Gross Margin", "Composure" leaked through.

**Changes to `_wbSkillQualityFilter()`:**
- Now calls `isSkillBlocklisted()` for admin blocklist
- Added personality traits to blocklist: composure, curiosity, integrity, empathy, humility, courage, patience, resilience, grit, tenacity, passion, enthusiasm
- Added fragment leaks: ashing, tform, monit, staf, hypot, analy, forma, techni, specif
- Added `metricBlocklist` regex: gross margin, net margin, EBITDA, revenue, profit, margin, quota, target, headcount, turnover, attrition, run rate, burn rate, CAGR, ARPU, ARR, MRR, NPS, CSAT, LTV, CAC, ROI

**Location:** `_wbSkillQualityFilter()` function, ~line 10573.

## For Replit: Stability / Performance / Security Review

### Performance Concerns

1. **Pass 6 seniority implication** iterates `Object.keys(domainKeywords)` × `userSkills.forEach()` for every unmatched gap skill. With 30 user skills and 25 JD skills, this is ~750 iterations per unresolved gap. Acceptable for current scale but worth monitoring.

2. **Pass 4 fuzzy subcategory resolution** iterates all keys in `_escoSkillToSubcategory` (~43K entries) when exact name not found. This runs per unmatched gap skill. Consider building a reverse index if this becomes a bottleneck.

3. **`buildEscoCategoryIndex()` user skill indexing** uses `.find()` on the 43K library array for each user skill. With 35 user skills, that's ~1.5M comparisons at load. Could optimize with a Map lookup if load time regresses.

4. **Admin blocklist** loads from Firestore on every auth, even for non-admin users. This is a single document read, so minimal cost.

### Security Review Points

1. **Admin blocklist** writes to `meta/skillBlocklist` — verify Firestore rules allow admin-only writes. Reads should be open (all users need the blocklist for parsing).

2. **`adminBlockSkill()`** calls `readOnlyGuard()` before writes. Verify this guard catches showcase mode and non-admin states.

3. **Gap chip ✕ button** uses `onclick` with `event.stopPropagation()` and `escapeHtml()`. String escaping looks correct but verify no injection path via skill names containing single quotes (they're escaped with `replace(/'/g, "\\\\'")` ).

4. **Domain keyword regexes** in pass 6 operate on `jobSkill.name` (raw from parsed JD) and `us.name` (from user profile). Both should be sanitized upstream but worth confirming.

### Stability Review Points

1. **`_stemEquiv` is a static object** — no risk of mutation, but pass 5 depends on it being complete. New stem pairs may be needed as skills library grows.

2. **Pass 6 checks `userData.roles`** — verify this array exists for all user states (new user, demo profile, showcase mode). Empty array should cause pass 6 to skip gracefully (seniority defaults to empty string, `if` block not entered).

3. **`matchType` variable** is declared once per `jobSkills.forEach` iteration and defaults to `'exact'`. Each pass overwrites it only if `found` is set. Stored in matched entries as `matchType` field. Verify downstream consumers (job detail rendering, network graph, PDF export) handle all five values: exact, sibling, concept, implied, and the default exact.

4. **`_adminSkillBlocklist`** is a global Set. If `loadAdminSkillBlocklist()` is called multiple times (re-auth), it rebuilds from Firestore each time. No append risk.

### Known Remaining Issues
1. **`quickScoreJob()`** (Find Jobs tab scoring) still uses the old hardcoded `commonSkillTerms` gap detection. Consider wiring it through `parseJobLocally` or at minimum applying the admin blocklist.
2. **matchType visual indicators** — the `matchType` field (sibling/concept/implied) is stored in match data but not yet surfaced in the job detail UI. Could show a small badge like "≈ related" on sibling matches vs "✓ exact" on direct matches.
3. **Pass 5 concept-root** only works for skills that decompose into >2-char words. Single-word skills like "Analytics" need the stem root "analy" to match "analysis". If stems aren't in `_stemEquiv`, the match fails silently.
4. **Pass 6 domain keywords** are hardcoded across 8 domains. As the platform expands to trades, healthcare, creative fields, more domains will be needed.
