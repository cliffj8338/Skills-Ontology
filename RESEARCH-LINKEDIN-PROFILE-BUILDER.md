## 1. LinkedIn Integration Options — Full Assessment

### Option A: Sign In with LinkedIn (OpenID Connect)
**What you get:** name, email, profile photo, locale — that's it.

**Technical reality:**
- Standard OAuth 2.0 + OIDC. Scopes: `openid`, `profile`, `email`
- Self-service approval through LinkedIn Developer Portal (Products tab → request access)
- Authorization code expires in **20 seconds** — callback must be fast
- Returns JWT `id_token` with claims: `sub`, `name`, `given_name`, `family_name`, `picture`, `email`, `email_verified`, `locale`

**Firebase Auth compatibility — PROBLEM:**
- Firebase OIDC expects discovery doc at `{issuer}/.well-known/openid-configuration`
- LinkedIn hosts it at `{issuer}/oauth/.well-known/openid-configuration`
- This means **Firebase's built-in OIDC provider won't work with LinkedIn**
- LinkedIn is not a native Firebase Auth provider (Google, Apple, Facebook are; LinkedIn is not)

**Workaround (proven pattern):**
1. Serverless function handles the OAuth flow manually (`/api/linkedin-callback.js`)
2. Exchanges auth code for tokens server-side using LinkedIn's token endpoint
3. Verifies JWT using LinkedIn's JWKS endpoint
4. Creates a Firebase Custom Token via Admin SDK (`auth().createCustomToken()`)
5. Returns custom token to client
6. Client calls `signInWithCustomToken(auth, token)` — now the user is in Firebase Auth

**Effort:** Medium. One new serverless function + LinkedIn app registration + 2 env vars.
**Value for Blueprint:** Low-medium. Gets auth + basic identity, but zero career data.

---

### Option B: LinkedIn API v2 (Full Profile Data)
**What you'd get:** Positions, skills, education, certifications, recommendations

**Reality check:**
- Requires LinkedIn Partner Program approval
- Only available to incorporated companies, not individual developers
- Pricing not publicly disclosed, determined per-application
- Approval timeline: weeks to months, often denied
- Even if approved, rate limits are strict

**Verdict: Dead path for now.** Revisit only if Blueprint reaches a scale where LinkedIn partnership makes business sense. Not viable for MVP or early traction.

---

### Option C: LinkedIn Profile PDF Export (Save to PDF)
**What you get:** Visually formatted PDF of a LinkedIn profile

**Format characteristics:**
- Generated by LinkedIn's "Save to PDF" feature (More → Save to PDF)
- English characters only display correctly
- Not available on mobile app (desktop web only)
- Contains: summary, experience, education, skills, certifications, volunteer work, recommendations
- Layout mimics LinkedIn's profile page — not structured data

**Parsing approach:**
- Extract text from PDF (already have PDF processing via AI proxy)
- Send to Claude for structured extraction
- Already have resume parsing via `/api/ai.js` — LinkedIn PDF is similar format

**Pros:** Works today, no API dependency, user-controlled
**Cons:** Extra user step (go to LinkedIn → download → upload), PDF format can be inconsistent, AI parsing adds latency (~10-15s) and cost (~$0.02-0.05 per parse)

---

### Option D: LinkedIn Data Archive (GDPR Export) ⭐ NEW FINDING
**What you get:** Structured CSV files with complete profile data

**How it works:**
1. User goes to LinkedIn → Settings & Privacy → Data Privacy → "Get a copy of your data"
2. Selects specific data categories (or full archive)
3. LinkedIn emails download link within ~15 minutes (partial) or 24h (full)
4. Download is a `.zip` containing CSV and JSON files

**Files included (confirmed):**
| File | Contents |
|------|----------|
| `Profile.csv` | First Name, Last Name, Headline, Summary, Industry, Location, Websites |
| `Positions.csv` | Company Name, Title, Description, Location, Started On, Finished On |
| `Skills.csv` | Skill names (flat list) |
| `Education.csv` | School Name, Degree, Start/End Date, Notes, Activities |
| `Certifications.csv` | Name, Authority, License Number, Start/End dates, URL |
| `Languages.csv` | Name, Proficiency |
| `Projects.csv` | Name, description, start/end dates |
| `Email Addresses.csv` | Email addresses on file |

**This is structured data. No AI parsing needed.** Just CSV parsing — PapaParse on the client or simple JS. Maps almost directly to Blueprint's `userData` schema.

**Pros:**
- Already structured (CSV), no AI parsing cost or latency
- Complete career data: positions, skills, education, certs, languages, projects
- Legal and sanctioned by LinkedIn (their own export tool)
- Free, no API key, no partner approval
- User has full control over their data

**Cons:**
- 15-minute wait for LinkedIn to prepare the archive
- Extra steps for user (Settings → Data Privacy → request → wait → download → upload)
- Some users may not know how to do this

**Mitigation for the wait:**
- User can start manual entry or resume upload immediately
- Later, when LinkedIn archive arrives, import to enrich/override
- Or: show a "while you wait" guide with screenshots of the LinkedIn export process

---

### Option E: Manual LinkedIn URL Scraping
**Verdict: Hard no.** Violates LinkedIn ToS, legally risky, technically fragile.

## 2. Recommended Integration Strategy

### Phase 1 (This Session): Resume/PDF Upload + LinkedIn Data Archive
**Zero new API dependencies. Maximum data capture.**

```
New user signs up (Google or email — existing auth)
  → Welcome wizard
  → Source selection:
      📄 Upload Resume (PDF/DOCX) — AI parses via existing /api/ai.js
      📦 Import LinkedIn Data (.zip) — CSV parse, zero AI cost
      🔗 Import LinkedIn PDF — AI parses (same as resume)
      ✏️ Start from scratch — guided manual entry
  → Review & confirm extracted data
  → Profile complete → Dashboard
```

**Why lead with this, not LinkedIn OAuth:**
- LinkedIn OIDC gives you name + email. Google Auth already gives you that.
- The career data (positions, skills, education) is the hard part and the valuable part.
- LinkedIn Data Archive gives you structured career data with zero API dependency.
- Resume upload gives you career data with existing infrastructure.
- Adding LinkedIn OAuth for auth is incremental value that can come in Phase 2.

### Phase 2 (Future): LinkedIn OAuth Sign-In
- Add "Sign in with LinkedIn" button alongside existing Google sign-in
- Requires: LinkedIn app registration, serverless callback, Firebase custom token
- Value: one more auth option, LinkedIn profile photo, LinkedIn user ID for future API access
- Not blocking for the profile builder goal

## 3. Technical Implementation Plan

### Profile Builder Wizard — Architecture

**State management:**
```javascript
const wizardState = {
  step: 1,          // 1: welcome, 2: source, 3: processing, 4: review, 5: complete
  source: null,     // 'resume', 'linkedin-zip', 'linkedin-pdf', 'manual'
  extractedData: null,
  confirmed: false
};
```

**Source → Parser mapping:**
| Source | Parser | Cost | Latency |
|--------|--------|------|---------|
| Resume PDF/DOCX | Claude via `/api/ai.js` | ~$0.02-0.05 | 10-15s |
| LinkedIn .zip | Client-side CSV parse (PapaParse) | $0 | <1s |
| LinkedIn PDF | Claude via `/api/ai.js` | ~$0.02-0.05 | 10-15s |
| Manual entry | None | $0 | User-paced |

**LinkedIn .zip parsing flow (client-side):**
```
1. User uploads .zip file
2. JSZip extracts in browser
3. Parse CSV files with PapaParse:
   - Profile.csv → userData.profile (name, headline, location, summary)
   - Positions.csv → userData.workHistory (title, company, dates, description)
   - Skills.csv → userData.skills (name, need AI for level/category inference)
   - Education.csv → userData.education
   - Certifications.csv → userData.certifications
4. For skills that need level/category: optional Claude call to infer
5. Map to Blueprint data structures
6. Show review screen
```

**CDN dependencies needed:**
- JSZip: `https://cdnjs.cloudflare.com/ajax/libs/jszip/3.10.1/jszip.min.js`
- PapaParse: `https://cdnjs.cloudflare.com/ajax/libs/PapaParse/5.4.1/papaparse.min.js`
  (or: both are small enough to inline if we want zero new deps)

**Resume/LinkedIn PDF parsing flow (existing):**
```
1. User uploads PDF/DOCX
2. Convert to base64
3. Send to /api/ai.js with structured extraction prompt
4. Claude returns JSON: { profile, skills, roles, workHistory, education, certifications }
5. Map to Blueprint data structures
6. Show review screen
```

### Data Mapping — LinkedIn CSV → Blueprint userData

```javascript
// Positions.csv → workHistory
{
  title: row['Title'],
  company: row['Company Name'],
  location: row['Location'],
  description: row['Description'],
  startDate: row['Started On'],    // "Jan 2020" or "2020" format
  endDate: row['Finished On'] || 'Present'
}

// Skills.csv → skills (need inference for level + category)
// LinkedIn just gives skill names. Blueprint needs: name, level, category, roles
// Two options:
//   A) Default all to level 3 (intermediate), let user adjust
//   B) Send skill list + positions to Claude for inference (~$0.01, 3-5s)
// Recommendation: Option B with fallback to A if API fails

// Education.csv → education  
{
  school: row['School Name'],
  degree: row['Degree Name'],
  field: row['Notes'],           // Often contains field of study
  startDate: row['Start Date'],
  endDate: row['End Date'],
  activities: row['Activities']
}

// Certifications.csv → certifications
{
  name: row['Name'],
  authority: row['Authority'],
  licenseNumber: row['License Number'],
  startDate: row['Started On'],
  endDate: row['Finished On'],
  url: row['Url']
}

// Profile.csv → profile
{
  name: `${row['First Name']} ${row['Last Name']}`,
  currentTitle: row['Headline'],
  location: row['Geo Location'],
  summary: row['Summary']
}
```

### Role Inference from Positions

Blueprint's core value is the role → skill mapping. When importing, we need to infer roles from work history.

**Approach:**
1. Extract unique job titles from Positions.csv
2. Send to Claude: "Given these job titles, suggest 3-7 role categories for this person's Blueprint"
3. Claude returns role suggestions with colors
4. User confirms/adjusts on review screen
5. Skills get auto-assigned to roles based on Claude's inference

This is a single ~$0.01-0.02 API call that transforms a flat resume into a structured Blueprint.

## 4. Firebase Considerations

### Auth — No Changes Needed for Phase 1
- Current Google + email/password auth covers the sign-up flow
- LinkedIn OAuth deferred to Phase 2
- No Firestore rule changes needed for profile builder (saving to existing `users/{uid}` doc)

### Firestore — Profile Data
- Profile builder saves to the same `userData` structure as manual entry
- `saveToFirestore()` already handles the full userData object
- No schema changes needed

### Security Note
- LinkedIn .zip is parsed entirely client-side — no PII touches our servers
- Resume PDF goes through `/api/ai.js` (Anthropic) — existing pattern, same security model
- Worth adding: file size validation (reject > 10MB uploads)

## 5. Known Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| LinkedIn changes CSV export format | Parser breaks | Version-detect CSV headers, fallback to AI parse |
| Users don't know how to export LinkedIn data | Low adoption of zip import | Clear in-app guide with screenshots, lead with resume upload |
| Claude API latency on resume parse | Poor UX | Show animated progress, cache results, optimize prompt |
| Large resume files | Upload failures | Client-side size check (10MB max), compress before send |
| Mobile file upload UX | Friction | Use native file picker, accept from cloud storage apps |
| LinkedIn .zip contains unexpected files | Parser confusion | Whitelist expected filenames, ignore others |
